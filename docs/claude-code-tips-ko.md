# Claude Code 45가지 팁 (한국어 요약)

원본: https://github.com/ykdojo/claude-code-tips

---

## Tip 0: 상태 표시줄 커스터마이즈
Claude Code 하단에 모델명, 현재 디렉토리, Git 브랜치, 미커밋 파일 수, 토큰 사용률 등을 표시하는 상태 표시줄을 설정할 수 있다. 컨텍스트 사용량을 실시간으로 확인하고, 마지막 메시지를 표시하여 작업 내용을 기억하는 데 유용하다. 10가지 색상 테마를 지원하며, 설정 스크립트는 원본 리포의 `scripts/context-bar.sh` 참조.

## Tip 1: 필수 슬래시 명령어 익히기
`/usage`로 요금 한도를 확인하고, `/chrome`으로 브라우저 통합을 토글하고, `/mcp`로 MCP 서버를 관리하고, `/stats`로 GitHub 스타일 활동 그래프를 볼 수 있다. `/clear`로 대화를 초기화하여 새로 시작할 수 있다. 터미널에서 `/`를 입력하면 전체 명령어 목록을 볼 수 있다.

## Tip 2: 음성으로 Claude Code와 대화하기
타이핑보다 음성 입력이 훨씬 빠르다. Mac에서는 superwhisper, MacWhisper, Super Voice Assistant(오픈소스) 등을 사용할 수 있다. 로컬 모델로도 충분하며, 전사 오류가 있어도 Claude가 의도를 잘 파악한다. 이어폰으로 속삭이면 사무실이나 비행기에서도 사용 가능하다.

## Tip 3: 큰 문제를 작은 문제로 분해하기
Claude Code가 복잡한 문제를 한번에 해결하지 못하면, 여러 작은 하위 문제로 분해하라. A에서 B로 직접 가는 대신 A->A1->A2->A3->B로 단계별로 진행한다. 소프트웨어 엔지니어링의 기본 원칙과 동일하며, 문제 해결 능력은 에이전트 코딩 시대에도 여전히 핵심 역량이다.

## Tip 4: Git과 GitHub CLI를 프로처럼 활용하기
커밋, 브랜치, 풀, 푸시 등 Git 작업을 Claude에게 맡기면 커밋 메시지를 직접 쓸 필요가 없다. pull은 자동 허용하되 push는 위험하므로 수동 확인을 권장한다. `gh` CLI로 드래프트 PR을 만들면 리뷰 전 안전하게 확인할 수 있으며, GraphQL 쿼리도 가능하다. `~/.claude/settings.json`에서 Co-Authored-By 표기를 비활성화할 수도 있다.

## Tip 5: AI 컨텍스트는 우유처럼 신선하고 농축될수록 좋다
새 대화를 시작하면 이전 컨텍스트의 복잡성이 없어 성능이 가장 좋다. 대화가 길어질수록 성능이 저하되므로, 새로운 주제마다 새 대화를 시작하거나 성능이 떨어지기 시작하면 새로 시작하라.

## Tip 6: 터미널에서 출력 가져오기
`/copy`로 마지막 응답을 클립보드에 복사하거나, `pbcopy`로 직접 클립보드에 보내거나, 파일로 저장한 후 VS Code에서 열 수 있다. URL을 브라우저에서 열거나 GitHub Desktop에서 리포를 여는 것도 가능하다. PR 설명을 편집할 때는 로컬 파일로 먼저 작업한 후 다시 붙여넣는 방식이 효과적이다.

## Tip 7: 터미널 별칭(alias) 설정하기
자주 사용하는 명령어에 짧은 별칭을 설정하라. 예: `c`=claude, `ch`=claude --chrome, `gb`=github, `co`=code. `~/.zshrc`나 `~/.bashrc`에 추가한다. `c -c`로 마지막 대화를 이어가거나 `c -r`로 최근 대화 목록에서 선택할 수 있다.

## Tip 8: 컨텍스트를 능동적으로 압축하기
`/compact` 명령으로 대화를 요약하여 컨텍스트 공간을 확보할 수 있지만, 더 좋은 방법은 핸드오프 문서(HANDOFF.md)를 작성하는 것이다. 현재까지의 작업 상태, 시도한 것, 성공/실패 내용, 다음 단계를 정리한 후 새 대화를 시작하면 된다. `/handoff` 슬래시 명령이나 plan 모드(`/plan` 또는 Shift+Tab)를 활용할 수도 있다.

## Tip 9: 자율 작업을 위한 쓰기-테스트 주기 완성하기
Claude Code가 자율적으로 작업하려면 결과를 검증할 방법이 필요하다. tmux를 사용하여 세션을 시작하고, 명령을 보내고, 출력을 캡처하고, 검증하는 패턴이 효과적이다. `git bisect` 자동화나 테스트 작성에 활용할 수 있다. 웹 앱은 Playwright MCP나 Chrome 통합을 사용하되, 대부분의 경우 Playwright가 더 안정적이다.

## Tip 10: Cmd+A / Ctrl+A 활용하기
Claude Code가 직접 접근할 수 없는 페이지(비공개 페이지, Reddit 등)의 내용을 전체 선택(Cmd+A/Ctrl+A)하여 복사한 후 Claude Code에 붙여넣으면 된다. 터미널 출력에도 동일하게 적용된다. Gmail 스레드의 경우 "모두 인쇄"로 미리보기를 열면 깔끔하게 전체 선택할 수 있다.

## Tip 11: 차단된 사이트에 Gemini CLI를 대체 수단으로 사용하기
Claude Code의 WebFetch가 접근할 수 없는 사이트(예: Reddit)에는 Gemini CLI를 우회 수단으로 사용할 수 있다. tmux 패턴(Tip 9)으로 Gemini에 명령을 보내고 출력을 캡처하는 스킬을 만들면 된다. 속도는 느리지만 결과물의 품질이 좋다. `~/.claude/skills/reddit-fetch/SKILL.md`에 스킬 파일을 배치한다.

## Tip 12: 자신의 워크플로우에 투자하기
커스텀 음성 전사 앱, 상태 표시줄, 시스템 프롬프트 최적화 등 자신만의 도구를 구축하라. 반드시 과하게 할 필요는 없으며, CLAUDE.md를 잘 관리하고 핵심 기능과 팁을 배우는 것만으로도 충분하다. 도구에 대한 투자는 곧 생산성 향상으로 이어진다.

## Tip 13: 대화 기록 검색하기
대화 기록은 `~/.claude/projects/`에 로컬로 저장되며, 프로젝트 경로를 기반으로 폴더명이 정해진다. bash 명령으로 `.jsonl` 파일을 검색하거나, Claude Code에 직접 "오늘 X에 대해 뭘 이야기했지?"라고 물어보면 된다. 코드는 원본 참조.

## Tip 14: 터미널 탭으로 멀티태스킹하기
여러 Claude Code 인스턴스를 실행할 때는 조직화가 핵심이다. "캐스케이드" 방식으로 새 작업마다 오른쪽에 새 탭을 열고 왼쪽에서 오른쪽으로 순회한다. 동시에 최대 3~4개 작업에 집중하는 것을 권장한다.

## Tip 15: 시스템 프롬프트 슬림화
Claude Code의 시스템 프롬프트와 도구 정의는 약 19k 토큰(컨텍스트의 ~10%)을 차지한다. 패치 시스템으로 이를 약 9k 토큰으로 줄여 ~50% 절약할 수 있다. 장황한 예시와 중복 텍스트를 제거하면서 핵심 지시사항은 유지한다. 자동 업데이트를 비활성화하려면 `DISABLE_AUTOUPDATER: "1"`을 설정하라. MCP 도구는 `ENABLE_TOOL_SEARCH: "true"`로 지연 로딩하여 추가 절약이 가능하다.

## Tip 16: Git worktree로 병렬 브랜치 작업
같은 프로젝트에서 여러 작업을 동시에 할 때 Git worktree를 사용하면 충돌 없이 다른 브랜치를 다른 디렉토리에서 동시에 작업할 수 있다. Claude Code에 worktree 생성을 요청하면 구체적인 구문을 몰라도 된다. 멀티태스킹 캐스케이드 방식과 결합하면 효과적이다.

## Tip 17: 긴 작업에 수동 지수 백오프 적용하기
Docker 빌드나 GitHub CI 같은 장시간 작업을 기다릴 때, Claude Code에 점점 늘어나는 간격(1분, 2분, 4분...)으로 상태를 확인하도록 요청하라. `gh run watch`보다 토큰 효율적이며, 전용 대기 명령이 없는 상황에서도 활용 가능한 범용 기법이다.

## Tip 18: Claude Code를 글쓰기 보조 도구로 활용하기
먼저 맥락과 상세한 지시를 음성으로 전달하여 초안을 얻고, 그 후 한 줄씩 함께 검토하면서 수정한다. "이 줄은 좋다", "이 부분을 저쪽으로 옮기자", "이런 식으로 바꾸자" 등 구체적으로 피드백한다. 터미널과 코드 에디터를 나란히 놓고 작업하면 효과적이다.

## Tip 19: 마크다운이 최고다
AI 시대에 문서 작성의 가장 효율적인 방법은 마크다운이다. 블로그 포스트든 LinkedIn 게시글이든 Claude Code와 대화하며 마크다운으로 저장할 수 있다. 마크다운을 받지 않는 플랫폼에는 Notion에 먼저 붙여넣은 후 복사하면 된다. Cmd+Shift+V로 서식 없이 붙여넣기도 가능하다.

## Tip 20: Notion으로 링크가 포함된 텍스트 붙여넣기
Slack 등에서 링크가 포함된 텍스트를 Claude Code에 직접 붙여넣으면 링크가 사라진다. Notion 문서에 먼저 붙여넣은 후 다시 복사하면 마크다운 형태의 링크가 보존된다.

## Tip 21: 컨테이너에서 장시간/위험한 작업 실행하기
`--dangerously-skip-permissions`를 호스트에서 직접 사용하는 대신 컨테이너에서 실행하면 안전하다. 연구, 실험, 시스템 프롬프트 패치 작업 등에 유용하다. 로컬 Claude Code가 tmux를 통해 컨테이너 내 Claude Code를 제어하는 "워커" 패턴도 가능하다. SafeClaw 도구로 컨테이너 세션을 쉽게 관리할 수 있다.

## Tip 22: Claude Code를 잘 쓰는 최고의 방법은 직접 써보는 것
실제로 많이 사용하는 것이 가장 좋은 학습 방법이다. "10,000시간 법칙" 대신 "10억 토큰 법칙"을 생각하라. 보조 자료(영상, 책, 팁)도 도움이 되지만, 직접 사용하며 직관을 키우는 것이 핵심이다.

## Tip 23: 대화 복제(Clone)와 반복제(Half-Clone)
대화의 특정 지점에서 다른 방향을 시도하고 싶을 때 대화를 복제할 수 있다. 내장 기능으로 `/fork`나 `--fork-session` 플래그를 사용하거나, 커스텀 `clone-conversation` 스크립트를 사용한다. 대화가 길어졌을 때는 `half-clone`으로 후반부만 유지하여 토큰을 절약한다. 컨텍스트 85% 초과 시 자동으로 half-clone을 제안하는 훅도 설정 가능하다.

## Tip 24: realpath로 절대 경로 얻기
다른 폴더의 파일을 Claude Code에 알려줄 때 `realpath some/relative/path`로 절대 경로를 얻어 전달하라.

## Tip 25: CLAUDE.md vs Skills vs Slash Commands vs Plugins 이해하기
**CLAUDE.md**는 모든 대화에 자동 로드되는 기본 프롬프트이다. **Skills**는 필요할 때만 로드되므로 토큰 효율적이며, Claude가 자동으로 또는 사용자가 수동으로 호출한다. **Slash Commands**는 사용자가 정확한 타이밍에 호출하는 용도이며, Skills와 기능적으로 유사하여 현재는 통합되었다. **Plugins**는 Skills, Slash Commands, 에이전트, 훅, MCP 서버를 하나로 묶는 패키지 형태이다.

## Tip 26: 인터랙티브 PR 리뷰
Claude Code에 `gh` 명령으로 PR 정보를 가져오게 한 후 대화형으로 리뷰할 수 있다. 전체 개요부터 파일별 상세 리뷰까지, 원하는 속도와 깊이로 진행 가능하다. 일회성 리뷰 도구와 달리 대화를 통해 깊이 파고들 수 있다는 것이 핵심 차이점이다.

## Tip 27: Claude Code를 리서치 도구로 활용하기
GitHub Actions 실패 원인 분석, Reddit 감성/시장 분석, 코드베이스 탐색, 공개 정보 검색 등 다양한 연구에 활용 가능하다. 핵심은 적절한 정보 접근 방법을 제공하는 것이다: `gh` CLI, 컨테이너, Gemini CLI(Reddit), Slack MCP, Cmd+A 방식, Playwright MCP 등.

## Tip 28: 출력 검증 방법 마스터하기
테스트 작성, Claude Code UI에서 직접 코드 확인, GitHub Desktop 같은 비주얼 Git 클라이언트 사용, 드래프트 PR 생성 후 확인 등 다양한 검증 방법이 있다. "정말 확실해? 다시 확인해봐"라고 요청하거나, "모든 주장을 이중 확인하고 검증 결과를 표로 만들어줘"라고 하면 효과적이다.

## Tip 29: Claude Code를 DevOps 엔지니어로 활용하기
GitHub Actions CI 실패를 Claude Code에 맡기면 로그를 분석하고 근본 원인을 찾아준다. 표면적 답변을 넘어 "특정 커밋/PR 때문인지, 플래키 이슈인지" 계속 물으면 깊이 파고든다. `/gha <url>` 슬래시 명령으로 자동 조사가 가능하다. 문제 확인 후 드래프트 PR을 만들어 수정까지 진행할 수 있다.

## Tip 30: CLAUDE.md를 단순하게 유지하고 주기적으로 검토하기
CLAUDE.md 없이 시작해도 된다. 같은 말을 반복하게 되면 그때 추가하라. `#` 기호나 Claude Code에 직접 요청하여 편집할 수 있다. 시간이 지나면 지시사항이 구식이 될 수 있으므로 주기적으로 검토하라. `review-claudemd` 스킬로 최근 대화를 분석하여 개선점을 제안받을 수 있다.

## Tip 31: Claude Code를 범용 인터페이스로 활용하기
Claude Code는 코딩뿐 아니라 동영상 편집(ffmpeg), 오디오 전사(Whisper), 데이터 분석(Python/JS), 저장 공간 정리 등 컴퓨터의 모든 디지털 작업에 활용할 수 있다. 터미널 탭을 여러 개 열면 마치 여러 개의 두뇌를 동시에 가동하는 것과 같다. 지루하고 반복적인 작업을 위임하기에 특히 좋다.

## Tip 32: 적절한 추상화 수준 선택하기
바이브 코딩(코드를 안 봄)부터 한 줄씩 검토하는 것까지 스펙트럼이 있다. 일회성 프로젝트나 비핵심 코드는 바이브 코딩으로 충분하지만, 중요한 코드는 파일 구조, 함수, 개별 코드 라인까지 깊이 확인해야 한다. 상황에 따라 유연하게 수준을 조절하라.

## Tip 33: 허용된 명령어 감사하기
`.claude/settings.json`의 허용된 명령어에 `rm -rf`, `sudo`, `chmod 777` 같은 위험한 패턴이 없는지 주기적으로 확인하라. `cc-safe` CLI 도구(`npx cc-safe .`)로 자동 스캔할 수 있다. 실제로 홈 디렉토리가 삭제된 사례가 있으므로 주의가 필요하다.

## Tip 34: 테스트를 많이 작성하고 TDD를 활용하기
코드베이스가 커질수록 테스트가 중요하다. AI가 자기 코드를 테스트할 수 없다는 말은 사실이 아니다. TDD 방식(테스트 먼저 작성 -> 실패 확인 -> 커밋 -> 구현 -> 통과)이 Claude Code와 특히 잘 맞는다. 테스트가 단순히 true를 반환하지 않는지 직접 확인하는 것도 중요하다.

## Tip 35: 미지의 영역에서 더 과감하게, 반복적 문제 해결
Claude Code와 함께라면 익숙하지 않은 기술(React, Rust 등)에서도 반복적으로 문제를 해결할 수 있다. 초기 솔루션이 불충분하면 속도를 줄이고, "이 줄이 정확히 무슨 뜻이지?" 같은 질문을 하며 추상화 수준과 속도를 조절하라. 막다른 길에 부딪히면 다른 방향을 시도하는 협력적이고 반복적인 과정이다.

## Tip 36: 배경에서 bash 명령과 서브에이전트 실행하기
장시간 실행되는 bash 명령에 Ctrl+B를 누르면 백그라운드로 전환된다. Claude Code가 나중에 BashOutput 도구로 확인할 수 있다. 서브에이전트도 백그라운드로 실행 가능하며, 대규모 코드베이스 분석 시 여러 서브에이전트를 병렬로 생성하여 각 부분을 담당하게 할 수 있다. 서브에이전트의 수, 전경/배경, 모델(Opus/Sonnet/Haiku)을 지정할 수 있다.

## Tip 37: 개인화 소프트웨어의 시대
AI 덕분에 개인 용도의 맞춤 소프트웨어를 쉽게 만들 수 있는 시대가 왔다. 커스텀 전사 도구, 데이터 시각화, Slack CLI 등을 몇 시간 만에 구축할 수 있다. Docker 호환 문제로 기존 도구가 안 되면 Claude Code에 직접 CLI를 만들어달라고 하면 된다. HTML 슬라이드 덱 같은 프레젠테이션도 가능하다.

## Tip 38: 입력창 탐색 및 편집
Claude Code 입력창은 터미널/readline 단축키를 지원한다. `Ctrl+A`(줄 시작), `Ctrl+E`(줄 끝), `Ctrl+W`(이전 단어 삭제), `Ctrl+U`(줄 시작까지 삭제), `Ctrl+K`(줄 끝까지 삭제), `Ctrl+G`(외부 편집기로 열기). 줄바꿈은 `\` + Enter로 입력한다. 이미지는 Mac/Linux에서 `Ctrl+V`, Windows에서 `Alt+V`로 붙여넣는다.

## Tip 39: 계획에 시간을 투자하되 빠르게 프로토타입하기
Claude Code가 무엇을, 어떻게 만들지 알 수 있도록 기술 선택, 프로젝트 구조, 파일 구성 등 고수준 의사결정을 초기에 하라. 프로토타이핑으로 기술 적합성을 빠르게 검증할 수 있다. plan 모드(Shift+Tab)를 활용하거나 코드 작성 전 계획을 먼저 세우도록 요청하라.

## Tip 40: 과도하게 복잡한 코드 단순화하기
Claude Code는 요청하지 않은 변경을 하거나 과도한 코드를 작성하는 경향이 있다. "왜 이 변경을 했어?", "왜 이 줄을 추가했어?"라고 질문하고 단순화를 요청하라. AI가 쓴 코드를 이해할 수 없다는 것은 충분히 질문하지 않기 때문이다. 산문에서도 마지막 문단에 이전 내용을 요약하는 반복적 패턴에 주의하라.

## Tip 41: 자동화의 자동화
반복되는 작업을 CLAUDE.md에 넣고, 반복되는 명령은 스킬이나 스크립트로 만들어라. 타이핑 자동화를 위해 음성 전사 앱을 만든 것처럼, 반복 패턴이 보이면 자동화 방법을 찾는 것이 생산성과 재미 모두를 높인다.

## Tip 42: 지식을 공유하고 기여하기
팁을 공유하면 피드백을 통해 새로운 것을 배울 수 있다. Claude Code 리포에 이슈를 보내면 반영될 수도 있다(실제로 `/permissions` 기능 개선이 반영됨). 공유는 일방적이 아니라 양방향 학습 과정이다.

## Tip 43: 계속 배우기
Claude Code에 자체 기능에 대해 직접 질문하거나, `/release-notes`로 최신 기능을 확인하거나, r/ClaudeAI 서브레딧에서 다른 사용자의 워크플로우를 배우라. Anthropic DevRel인 Ado(@adocomplete)의 X 게시물도 유용하다.

## Tip 44: dx 플러그인 설치하기
이 리포는 `dx`라는 Claude Code 플러그인이기도 하다. `/dx:gha`(GitHub Actions 분석), `/dx:handoff`(핸드오프 문서), `/dx:clone`(대화 복제), `/dx:half-clone`(반복제), `/dx:reddit-fetch`(Reddit 콘텐츠), `/dx:review-claudemd`(CLAUDE.md 리뷰) 등을 제공한다. 두 줄의 명령으로 설치 가능하다. 코드는 원본 참조.

## Tip 45: 빠른 설정 스크립트
리포의 여러 권장 설정을 한번에 적용하는 설정 스크립트가 있다. DX 플러그인, cc-safe, 상태 표시줄, 자동 업데이트 비활성화, MCP 지연 로딩, 셸 별칭 등을 선택적으로 설치할 수 있다. 코드는 원본 참조.

## 신규 사용자를 위한 10가지 팁

원본: `content/10-tips-for-newer-users.md`

### 1. 터미널 vs VS Code vs Desktop vs Cowork
터미널 버전이 가장 기능이 풍부하고 개발이 활발하다. 가능하면 터미널에서 시작하는 것을 권장한다.

### 2. 특정 버전 설치하기
최신 버전이 불안정할 수 있으므로 안정된 특정 버전(예: 2.1.19)을 설치하라. `curl -fsSL https://claude.ai/install.sh | bash -s 2.1.19`로 설치 가능하다.

### 3. 중요 파일 백업과 버전 관리 사용
Claude Code가 실수할 수 있으므로 중요한 파일은 반드시 백업하고 Git으로 버전 관리하라. 롤백이 가능해야 안전하다.

### 4. 출력 테스트를 잘 하기
AI는 미묘한 버그를 도입할 수 있다. Claude Code가 작성한 테스트를 포함하여 출력을 철저히 검증하되, 테스트 자체가 유효한지도 확인하라.

### 5. 반복 지시를 CLAUDE.md에 넣기
같은 말을 반복하게 되면 CLAUDE.md에 추가하라. 프로젝트별(`./CLAUDE.md`) 또는 전역(`~/.claude/CLAUDE.md`). 일관되지 않은 지시에는 훅(hook)을 설정할 수도 있다.

### 6. 브라우저 통합 설정하기
Playwright MCP(`claude mcp add -s user playwright npx @playwright/mcp@latest`)가 대부분의 작업에 더 적합하다. 로그인 상태가 필요할 때만 `/chrome`을 사용하라.

### 7. 출력을 빠르게 리뷰하기
GitHub Desktop의 diff 뷰로 변경 사항을 쉽게 확인할 수 있다. 드래프트 PR을 만들어 리뷰한 후 실제 PR로 전환하는 방법도 효과적이다.

### 8. 리서치 -> 계획 -> 실행 -> 테스트
문제를 먼저 이해하고(리서치), Claude Code로 계획을 세우고(plan 모드), 코드를 작성하고(실행), 작동을 확인하라(테스트). 이 4단계 순서를 지키면 효과적이다.

### 9. 각 대화를 짧게 유지하기
새 대화일수록 성능이 좋다. 새로운 주제마다 새 대화를 시작하고, 긴 대화 하나보다 짧고 집중된 여러 대화가 낫다.

### 10. 여러 세션을 동시에 운용하기
2~3개의 터미널 탭에서 동시에 Claude Code를 실행하라. 캐스케이드 방식으로 새 작업은 오른쪽 탭에 열고 왼쪽에서 오른쪽으로 순회한다. 동시에 3~4개 이상은 관리가 어렵다.

### 11. (보너스) Git worktree로 병렬 브랜치 작업
같은 프로젝트에서 충돌 없이 여러 브랜치를 동시에 작업하려면 Git worktree를 사용하라.

## 5가지 새로운 팁

원본: `content/5-new-tips.md`

### 1. `/copy` 명령
Claude의 마지막 응답을 마크다운 형태로 클립보드에 복사하는 가장 간단한 방법이다.

### 2. `/fork`와 `--fork-session`
내장 대화 포크 기능이다. 대화 중에는 `/fork`, CLI에서는 `--fork-session` 플래그를 사용한다. `--fs`로 단축할 수 있는 셸 함수를 설정하면 편리하다.

### 3. Plan 모드로 컨텍스트 핸드오프
Plan 모드(`/plan` 또는 Shift+Tab)에서 다음 에이전트에 필요한 모든 컨텍스트를 모은 후, Option 1("컨텍스트 초기화 후 자동 수락")을 선택하면 깨끗한 상태에서 새로 시작된다.

### 4. CLAUDE.md 주기적 검토
시간이 지나면 CLAUDE.md가 구식이 될 수 있다. `review-claudemd` 스킬로 최근 대화를 분석하여 개선점을 자동 제안받을 수 있다.

### 5. Parakeet으로 음성 전사
Super Voice Assistant(오픈소스)에 Parakeet v2 지원이 추가되었다. ~110x 실시간 속도에 1.69% 단어 오류율로, Claude Code 사용에 충분히 정확하다.

## Boris의 10가지 팁

원본: `content/boris-claude-code-tips.md` (Boris Cherny, Claude Code 창시자)

### 1. 병렬 작업을 더 많이 하라
3~5개의 Git worktree를 열고 각각 Claude 세션을 실행하라. 팀 내 가장 큰 생산성 향상 방법이다. za, zb, zc 같은 셸 별칭으로 worktree 간 전환을 빠르게 할 수 있다.

### 2. 복잡한 작업은 Plan 모드로 시작하라
계획에 에너지를 쏟으면 Claude가 한번에 구현할 수 있다. 문제가 생기면 밀어붙이지 말고 Plan 모드로 돌아가 재계획하라. 다른 Claude에 계획을 리뷰시키는 사람도 있다.

### 3. CLAUDE.md에 투자하라
교정할 때마다 "CLAUDE.md를 업데이트해서 이 실수를 반복하지 마"라고 지시하라. Claude는 자기 자신을 위한 규칙을 놀라울 만큼 잘 작성한다. 실수율이 눈에 띄게 줄어들 때까지 반복하라.

### 4. 자신만의 스킬을 만들어 Git에 커밋하라
하루에 한 번 이상 하는 작업은 스킬이나 슬래시 명령으로 만들라. 팀 예시: 중복 코드 찾기(`/techdebt`), Slack/GDrive/Asana/GitHub 통합 컨텍스트 명령, dbt 모델 작성 에이전트.

### 5. Claude가 대부분의 버그를 스스로 수정한다
Slack 버그 스레드를 붙여넣고 "고쳐"라고만 하거나, "실패하는 CI 테스트를 고쳐"라고 지시하라. 방법을 세세하게 지시하지 마라. Docker 로그를 보여주며 분산 시스템 문제도 해결 가능하다.

### 6. 프롬프팅 수준을 높여라
"이 변경사항을 철저히 검토하고, 내가 테스트를 통과할 때까지 PR을 만들지 마", "지금까지 아는 것을 바탕으로 처음부터 다시 우아하게 구현해" 같은 도전적 프롬프트를 사용하라. 스펙을 상세하게 쓰고 모호성을 줄이면 출력 품질이 올라간다.

### 7. 터미널 및 환경 설정
Ghostty 터미널을 추천한다. `/statusline`으로 컨텍스트 사용량과 Git 브랜치를 표시하라. 터미널 탭을 색상으로 구분하라. 음성 받아쓰기를 사용하라(macOS에서 fn 두 번 누르기) - 타이핑보다 3배 빠르다.

### 8. 서브에이전트를 활용하라
"서브에이전트를 사용해"라고 말하면 Claude가 더 많은 계산 자원을 투입한다. 서브에이전트에 작업을 위임하면 메인 컨텍스트 윈도우가 깨끗하게 유지된다. 훅을 통해 안전한 권한 요청을 Opus 4.5로 자동 승인할 수도 있다.

### 9. 데이터 및 분석에 Claude 활용하기
`bq` CLI(또는 데이터베이스 CLI/MCP/API)와 함께 사용하여 지표를 수집하고 분석하라. Boris는 6개월 이상 SQL을 직접 작성하지 않았다고 한다.

### 10. Claude로 학습하기
`/config`에서 "Explanatory" 또는 "Learning" 출력 스타일을 활성화하면 변경의 이유를 설명해준다. 시각적 HTML 프레젠테이션, ASCII 코드베이스 다이어그램 생성, 간격 반복 학습 스킬 구축도 가능하다.

## Agentic Coding 스펙트럼: 바이브 코딩에서 고품질 소프트웨어 엔지니어링까지

원본: `content/spectrum-of-agentic-coding.md`

바이브 코딩과 전통적 소프트웨어 엔지니어링은 대립하는 것이 아니라, 하나의 스펙트럼 위에 있다. 4단계로 나뉜다:

### Level 1: 바이브 코딩
코드의 존재를 잊고 AI가 전부 처리하게 한다. 에러 로그도 읽지 않고 AI에 던진다. 버전 관리, 보안 없음. 일회성 프로젝트나 빠른 프로토타입에는 적합하지만 장기 유지보수는 불가능하다.

### Level 2: 규율 있는 에이전트 코딩
Git 사용, 파일 수준의 이해, 기본 보안 조치, 대부분의 경우에 작동하는 테스트. 구조가 생기기 시작한다.

### Level 3: 에이전트 소프트웨어 엔지니어링
프로덕션 수준에 근접한다. 검증된 테스트(AI/사람 작성), pre-commit 훅, CI 작업, 함수 수준의 이해, 수동 테스트, E2E 테스트, 원샷 AI 코드 리뷰, 회귀 검사를 포함한다.

### Level 4: 고품질 소프트웨어 엔지니어링
AI 보조 코딩이지만 스태프 엔지니어가 수작업한 것과 구별되지 않는 품질이다. 줄 단위 이해, 자기 반성 루프("정말 확실해? 다른 접근법은?"), 인터랙티브 AI 코드 리뷰, 심층 연구(Postgres vs Cassandra 등), AI 기반 품질 관리(백그라운드 테스트, 브라우저 UI 테스트, 멀티플랫폼), 가속 학습("왜 이렇게 작성했어?")을 포함한다.
