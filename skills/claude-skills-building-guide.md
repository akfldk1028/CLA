## 1. Skills란 무엇인가?

**정의:** 반복되는 작업 흐름을 한 번 정의해 지속적으로 재사용하도록 설계된 **워크플로 지식 패키지**

핵심 비유:
- **MCP** = 주방 (도구 제공)
- **Skills** = 레시피 (도구 사용 방법)

> MCP가 '무엇을 할 수 있는지'를 제공한다면, Skills는 '어떻게 해야 하는지'를 고정하는 역할

**빠른 시작:** 상위 2~3개의 명확한 워크플로가 있다면, **15~30분 내 첫 스킬 구축 가능**

## 2. 기본 개념 (Fundamentals)

### 기본 구성

```
my-skill/
├── SKILL.md          # 필수 - YAML frontmatter + Markdown 지침
├── scripts/          # 선택 - 자동화 스크립트
├── references/       # 선택 - 참조 문서
└── assets/           # 선택 - 관련 자산
```

### SKILL.md 구조

```yaml
---
name: my-awesome-skill
description: "스킬 설명 (1024자 이내)"
version: 1.0.0
triggers:
  - "트리거 표현 1"
  - "트리거 표현 2"
compatibility:
  - claude-code
  - claude-ai
---

# 스킬 본문 (Markdown)

## 단계별 지침
...
```

### 3대 설계 원칙

#### 1. Progressive Disclosure (점진적 정보 공개)

| 단계 | 내용 | 로딩 시점 |
|------|------|-----------|
| 1단계 | YAML frontmatter | **항상** 로드 |
| 2단계 | SKILL.md 본문 | **필요 시** 로드 |
| 3단계 | 연결 파일 (scripts, references) | **판단 필요 시** 로드 |

#### 2. Composability (조합 가능성)
- 여러 스킬 동시 로드 가능
- 상호 충돌 없이 설계해야 함

#### 3. Portability (이식성)
- Claude.ai, Claude Code, API 환경에서 **동일 동작**

## 3. 계획과 설계 (Planning & Design)

### 출발점: 구체적 사용 사례 2~3개 정의

각 사용 사례에 포함할 것:
- **사용자의 목표** - 무엇을 끝내고 싶은가?
- **트리거 문장** - 어떤 말로 시작하는가?
- **단계적 작업** - 어떤 순서로 진행하는가?
- **필요 도구** - 어떤 MCP 도구가 필요한가?
- **최종 결과 상태** - 완료 후 무엇이 달라지는가?

### 3가지 Skill 유형

| 유형 | 특징 | 예시 |
|------|------|------|
| **문서/자산 생성** | 일관된 품질, 스타일 가이드 내장 | 디자인, 코드, 문서 생성 |
| **워크플로 자동화** | 다단계 프로세스, 검증 지점 | 스프린트 계획, 온보딩 |
| **MCP 강화** | 도구 접근 + 전문 지식 결합 | 여러 MCP 순차 조합 |

### YAML frontmatter 작성 규칙

- **1024자 이내** 유지
- **XML 태그 금지**
- **구체적 표현과 용어** 포함 (사용자가 실제로 하는 말)
- description이 트리거 매칭의 핵심

### 성공 기준

**정량적:**
- 의도된 요청의 대부분에서 자동 트리거
- 스킬 사용 시 도구 호출 수 & 토큰 사용량 감소
- MCP 호출 실패 없는 완료

**정성적:**
- 사용자 지시 없이 자율 진행
- 반복 실행 시 일관된 결과
- 신규 사용자도 첫 시도에 성공

## 4. 테스트와 반복 개선 (Testing & Iteration)

### 테스트 수준 선택

| 수준 | 도구 | 적합한 상황 |
|------|------|------------|
| 수동 테스트 | Claude.ai | 빠른 반복, 초기 설계 검증 |
| 스크립트 기반 | Claude Code | 자동화, 회귀 테스트 |
| API 기반 | Anthropic API | 정량적 비교, 대규모 배포 |

**권장:** 하나의 어려운 작업에 집중해 반복 개선

### 3대 핵심 테스트 영역

#### 1. 트리거 테스트
- 적절한 상황에서만 로드되는가?
- 표현 변화에도 작동하는가?
- 무관한 요청에서는 로드 안 되는가?

#### 2. 기능 테스트
- 의도한 결과 생성 여부
- MCP 호출 성공
- 오류 처리
- 엣지 케이스 대응

#### 3. 성능 비교 테스트
- 메시지 왕복 횟수
- MCP 호출 실패 여부
- 총 토큰 사용량

### 피드백 기반 개선

| 증상 | 원인 | 해결 |
|------|------|------|
| 스킬이 자동 로드 안 됨 | 트리거 부족 | description에 구체적 표현 추가 |
| 무관한 곳에서도 로드 | 트리거 과다 | 범위 축소, 부정 트리거 추가 |
| 결과 일관성 부족 | 실행 문제 | 지침 명확화, 오류 처리 강화 |

## 5. 배포와 공유 (Distribution & Sharing)

### 개인 사용자 배포

1. Skill 폴더를 로컬에 다운로드
2. zip 압축 후 Claude.ai에 업로드
3. **Settings > Capabilities > Skills** 경로

### 조직 배포

- 관리자가 워크스페이스 전체 배포
- 자동 업데이트 지원
- 중앙 집중식 관리

### API 활용

```
/v1/skills 엔드포인트
container.skills 파라미터로 스킬 지정
Claude Console을 통한 버전 관리
```

### GitHub 배포 권장 사항

1. **README.md** 포함 (사람을 위한 설치 방법, 용도, 스크린샷)
   - 단, Skill 폴더 내부에는 README 불포함
2. MCP 문서와 연계 - Skill 결합 시 가치 명시
3. 단계별 설치 가이드 + 테스트 프롬프트 예시 제공

### 오픈 스탠다드

- MCP와 동일하게 **오픈 표준** 공개
- 플랫폼 독립적
- `compatibility` 필드로 환경 제약 명시 가능

## 6. 패턴과 문제 해결 (Patterns & Troubleshooting)

### 접근 방식 선택

| 접근법 | 대상 | 특징 |
|--------|------|------|
| **문제 중심** (Problem-first) | 일반 사용자 | 사용자가 결과를 말하면, 스킬이 도구 호출 순서 결정 |
| **도구 중심** (Tool-first) | 전문가 사용자 | 사용자가 MCP를 알고 있고, 스킬이 전문 지식 제공 |

### 5가지 핵심 패턴

#### 1. 순차 워크플로 오케스트레이션
- 정해진 순서의 여러 단계
- **적합:** 온보딩, 계정 생성

#### 2. 다중 MCP 협업
- 여러 서비스를 연속 사용
- **적합:** 디자인 > 저장 > 태스크 생성 흐름

#### 3. 반복 개선
- 초안 > 검증 > 수정 루프
- **적합:** 보고서, 문서 품질 개선

#### 4. 상황 인식 도구 선택
- 상황별 최적 도구 자동 선택
- **적합:** 파일 크기/타입 기반 판단

#### 5. 도메인 특화 지능
- 전문 지식 내재화
- **적합:** 금융, 컴플라이언스 등 전문 분야

### 문제 해결 가이드

| 문제 | 원인 | 해결책 |
|------|------|--------|
| 업로드 실패 | SKILL.md 파일명 오류, YAML 형식 | 파일명 정확히 확인, YAML 구분자(`---`) 확인 |
| 트리거 안 됨 | description이 추상적 | 실제 사용 표현 포함 |
| 과도한 트리거 | 범위 너무 넓음 | 부정 트리거 추가 |
| MCP 호출 실패 | 연결 또는 인증 문제 | 서버 상태, API 키 확인 |
| 지침 미준수 | 지침이 장황 또는 모호 | 핵심을 상단 배치, 검증 조건 명시 |
| 성능 저하 | 과도한 컨텍스트 | SKILL.md 크기 축소, 동시 스킬 수 제한 |

## 빠른 시작 체크리스트

- [ ] 상위 2~3개 반복 워크플로 식별
- [ ] 각 워크플로의 사용 사례 정의 (목표, 트리거, 단계, 도구, 결과)
- [ ] SKILL.md 작성 (YAML frontmatter + Markdown 지침)
- [ ] 트리거 테스트 (올바른 상황에서만 활성화되는지)
- [ ] 기능 테스트 (의도한 결과가 나오는지)
- [ ] 성능 테스트 (토큰 효율, MCP 호출 성공률)
- [ ] 배포 (로컬 / GitHub / 조직 워크스페이스)

## 핵심 요약

> "Skills는 프롬프트가 아니라 **지속적으로 축적되는 작업 자산**이다."

| 구분 | MCP | Skills |
|------|-----|--------|
| 역할 | 무엇을 **할 수 있는지** | 어떻게 **해야 하는지** |
| 비유 | 주방 (도구) | 레시피 (방법) |
| 제공 | 도구 접근 | 도메인 지식 + 워크플로 |
